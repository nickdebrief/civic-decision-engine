# CDE v12 Attachment Architecture

## Core Rule

Canonical record verification hashes remain unchanged.

Attachments are referenced artifacts. They are not canonical record fields, and
they must not alter the existing record hash computation. Each attachment has
its own independent SHA-256 hash, visibility state, redaction state, and
lifecycle metadata.

## Proposed Schema

Proposed table: `record_attachments`

```sql
CREATE TABLE record_attachments (
    id INTEGER PRIMARY KEY AUTOINCREMENT,

    reference TEXT NOT NULL,
    record_version INTEGER NOT NULL,
    attachment_version INTEGER NOT NULL DEFAULT 1,

    filename TEXT NOT NULL,
    stored_filename TEXT NOT NULL,
    storage_path TEXT NOT NULL,

    content_type TEXT NOT NULL,
    file_size_bytes INTEGER NOT NULL,
    sha256_hash TEXT NOT NULL,

    visibility TEXT NOT NULL CHECK (visibility IN ('private', 'public')),
    redaction_status TEXT NOT NULL DEFAULT 'none'
        CHECK (redaction_status IN ('none', 'redacted', 'withheld')),
    redaction_note TEXT,

    title TEXT,
    description TEXT,
    source_label TEXT,

    uploaded_at TEXT NOT NULL,
    uploaded_by TEXT,
    supersedes_attachment_id INTEGER,

    is_latest INTEGER NOT NULL DEFAULT 1,
    is_deleted INTEGER NOT NULL DEFAULT 0,

    FOREIGN KEY (reference, record_version)
        REFERENCES records(reference, version),
    FOREIGN KEY (supersedes_attachment_id)
        REFERENCES record_attachments(id)
);
```

Suggested indexes:

```sql
CREATE INDEX idx_record_attachments_reference
ON record_attachments(reference, record_version);

CREATE INDEX idx_record_attachments_public
ON record_attachments(reference, visibility, redaction_status, is_latest, is_deleted);

CREATE UNIQUE INDEX idx_record_attachments_version
ON record_attachments(reference, record_version, filename, attachment_version);
```

## Storage Layout

Attachment files should be stored outside the static asset mount:

```text
/data/attachments/{reference}/
```

Suggested layout:

```text
/data/attachments/
  Strike-LA-20260510-001/
    v1/
      attachments/
        attachment-42-v1-295181fe.pdf
        attachment-43-v1-aa1a2099.jpeg
```

Stored filenames must be generated by the system, not trusted from upload input.
Original filenames may be retained as metadata only.

## Immutability Rules

- Uploaded attachment bytes are immutable.
- Replacing an attachment creates a new `attachment_version`.
- Prior attachment versions remain stored unless removed by a deliberate admin
  retention policy.
- Attachment hashes are computed from raw bytes at upload time.
- Attachment hashes do not replace or alter record verification hashes.

## Visibility

Attachments may be:

- `private`: visible only through authenticated admin/internal access.
- `public`: eligible for public manifest listing and public serving, subject to
  redaction status.

Private attachments must never be reachable by guessing a public URL.

## Redaction Status

Recommended states:

- `none`: artifact is public as uploaded if visibility is public.
- `redacted`: artifact is a redacted public artifact, or has redaction metadata.
- `withheld`: artifact metadata or bytes are withheld from public serving.

Public serving should allow only explicit public-safe states. A conservative
first implementation should serve `public + none` and may later support
`public + redacted` when the stored artifact is itself the redacted copy.

## Independent Versioning

Attachment versioning is separate from record versioning.

Example:

- Record `Strike-LA-20260510-001`, version `1`
- Attachment `42`, attachment version `1`
- Attachment `42`, attachment version `2`, superseding version `1`

Changing an attachment version must not change the record version or record
verification hash.

## Manifest Expansion

Record manifests should be expanded additively:

```json
{
  "manifest_version": "1.1",
  "manifest_type": "civic_decision_engine_record",
  "reference": "Strike-LA-20260510-001",
  "verification_hash": "...",
  "canonical_fields": {},
  "attachments": []
}
```

When public attachments exist:

```json
{
  "attachments": [
    {
      "attachment_id": 42,
      "attachment_version": 1,
      "filename": "council-letter-redacted.pdf",
      "content_type": "application/pdf",
      "file_size_bytes": 184322,
      "sha256_hash": "attachment-sha256...",
      "visibility": "public",
      "redaction_status": "redacted",
      "redaction_note": "Personal contact details removed.",
      "title": "Council response letter",
      "description": "Redacted public copy of correspondence.",
      "source_label": "Uploaded supporting evidence",
      "uploaded_at": "2026-05-27T10:00:00Z",
      "download_url": "https://civic-decision-engine-production.up.railway.app/records/Strike-LA-20260510-001/attachments/42"
    }
  ],
  "attachment_integrity_note": {
    "canonical_record_hash_unchanged": true,
    "attachment_hashes_independent": true
  }
}
```

Public manifests must not expose private storage paths.

## Public Serving Rules

Public attachment serving should be controlled by application routes, not direct
static file serving.

Candidate route:

```text
GET /records/{reference}/attachments/{attachment_id}
```

Serve only when:

- the attachment belongs to the referenced record;
- `visibility = public`;
- `is_latest = 1`;
- `is_deleted = 0`;
- redaction status is public-safe under the active policy.

Recommended headers:

- `X-Content-Type-Options: nosniff`
- conservative `Content-Disposition`
- explicit `Content-Type` from validated metadata

## Admin Upload Direction

Upload should be admin/API only. No public upload UI should be added in v12.

Candidate route:

```text
POST /api/admin/records/{reference}/attachments
```

Inputs:

- multipart file
- optional `record_version`, defaulting to latest
- `visibility`
- `redaction_status`
- `title`
- `description`
- `source_label`
- `redaction_note`

Metadata-only updates may be handled separately:

```text
PATCH /api/admin/attachments/{attachment_id}
```

Replacing bytes should create a new version:

```text
POST /api/admin/attachments/{attachment_id}/versions
```

## Admin Authentication Requirement

Admin attachment routes must not be implemented until an explicit admin
authentication mechanism exists.

Preferred v12 direction:

- store a static admin token in Railway/environment as `CDE_ADMIN_TOKEN`;
- require admin requests to send `X-CDE-Admin-Token`;
- keep admin attachment routes disabled when `CDE_ADMIN_TOKEN` is unset;
- never log the token or include it in error responses;
- require the token for upload, metadata update, and attachment version routes.

This token protects admin attachment operations only. Public attachment serving
remains governed separately by visibility and redaction rules. There must be no
public upload route.

## Citation Implications

The verification page citation remains a citation to the canonical record.

Attachment citations should be separate and include the attachment SHA-256 hash:

```text
Civic Decision Engine. (2026). Attachment to Strike-LA-20260510-001:
Council response letter. SHA-256: [attachment hash]. [URL]
```

Public pages should clearly distinguish:

- the record verification hash, which verifies canonical record fields;
- attachment hashes, which verify artifact bytes.

## Security And Privacy Risks

Primary risks:

- accidental public exposure of private attachments;
- path traversal through filenames;
- MIME spoofing and unsafe content sniffing;
- oversized uploads;
- malware hosting;
- incomplete or misleading redaction;
- private or withheld files remaining accessible by direct URL;
- metadata leaking private information;
- caches retaining files after visibility changes;
- attachment IDs revealing private attachment existence;
- admin upload endpoints without sufficient authentication and authorization.

Mitigations:

- store files outside `/static`;
- serve files only through checked routes;
- sanitize original filenames and generate stored filenames;
- enforce upload size limits;
- use a MIME allowlist;
- require admin authentication for upload and private access;
- never expose filesystem paths;
- audit upload, versioning, visibility, and redaction changes;
- test direct access after visibility changes.

## Staged Implementation Order

1. Add schema migration and storage helpers.
2. Add attachment SHA-256 helper and immutable path builder.
3. Add admin/local upload route behind authentication.
4. Add manifest `attachments: []` expansion.
5. Add public attachment metadata and public serving rules.
6. Add independent attachment versioning.
7. Add documentation and operational guidance.

Each stage must prove that canonical record verification hashes remain unchanged.

## Migration Strategy

Migration should be additive:

- create `record_attachments` if missing;
- create indexes if missing;
- do not mutate `records`;
- do not modify verification hash logic;
- do not backfill attachments automatically.

Suggested migration file:

```text
api/static/migrations/add_record_attachments_table.py
```

Deployment order:

1. deploy schema;
2. deploy code that tolerates zero attachments;
3. add manifest `attachments: []`;
4. enable admin upload after privacy tests pass;
5. enable public serving after route access tests pass.

## Explicit v12 Non-Goals

Out of scope for v12:

- public upload UI;
- unauthenticated upload API;
- PDF text extraction;
- OCR;
- semantic indexing of attachment content;
- attachment embeddings;
- attachment text search;
- changing canonical record hash logic;
- adding attachment hashes to canonical record hashes;
- exposing private attachment discovery publicly;
- automatic redaction;
- frontend attachment management UI;
- external storage providers unless separately approved.
