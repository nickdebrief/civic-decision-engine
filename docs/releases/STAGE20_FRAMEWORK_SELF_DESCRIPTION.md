# Stage 20 - Framework Self-Description & Reflexive Methodology

## Purpose

Stage 20 enables the Civic Record Evaluation Framework to describe its existing
identity, purpose, scope, guarantees, constraints, reasoning architecture, and
methodology through deterministic declared outputs.

Stage 20 describes only implemented framework properties. It performs no new
record analysis and creates no methodology, identity, rules, conditions,
classifications, findings, recommendations, or predictions.

## Relationship to CREF Methodology

CREF evaluates what visible civic records and existing record-derived outputs
can support without undocumented assumptions. Stage 20 makes that methodology
inspectable at the framework level. It reports what CREF is, how its implemented
layers are arranged, what they guarantee, and where their declared boundaries
remain.

## Framework Version

The production repository does not declare a separate semantic methodology
version. Stage 20 therefore reports `CREF Stage 20`, the implemented framework
coverage represented by this self-description, rather than inventing a new
release number.

## Stage 20A - Framework Identity

The deterministic identity is:

- Framework Name: Civic Record Evaluation Framework
- Framework Acronym: CREF
- Framework Version: CREF Stage 20
- Framework Type: Deterministic Record Evaluation Framework
- Framework Purpose Label: Structured Evaluation of Visible Record Data

## Stage 20B - Purpose Description

The purpose description states why CREF exists, what it evaluates, what it
produces, and its intended operational role. These descriptions are fixed from
the implemented visible-record methodology and do not introduce a new purpose.

## Stage 20C - Scope Description

Declared inputs are visible record data, metadata, relationships, conditions,
and existing framework outputs.

Declared outputs include conditions, trajectories, administrative outputs,
record evolution classifications, determination traces, rule citations,
evidence attribution, explainability outputs, and certification outputs.

Operational scope remains limited to deterministic processing, record-derived
outputs, and no external truth determination.

## Stage 20D - Framework Guarantees

Stage 20 reports eight implemented guarantees:

1. Deterministic Execution
2. Traceability
3. Explainability
4. Evidence Attribution
5. Rule Visibility
6. Reproducibility
7. Boundary Transparency
8. Inspectability

Each guarantee includes a description, implemented basis, and availability
state. These are guarantees about framework behavior and visibility, not about
truth or underlying facts.

## Stage 20E - Framework Constraints

Stage 20 displays sixteen declared constraints. CREF does not determine truth,
liability, intent, blame, factual correctness, legal sufficiency, or real-world
evidential sufficiency. It does not validate evidence, create evidence, create
rules, create conditions, create framework classifications, or modify records.
Stage 20 also does not introduce new reasoning, perform self-improvement, or
change framework behaviour.

Each constraint is displayed with its description and methodology limitation
basis.

2. Conditions Layer

4. Trajectory Classification Layer
5. Administrative Evaluation Layer
6. Record Evolution Layer
7. Determination Trace Layer
8. Rule Citation Layer
9. Evidence Attribution Layer

Every entry identifies its implemented source stages and output state. No stage

## Stage 20G - Reflexive Methodology Description

The reflexive description records what CREF does, what it does not do, how it
reasons, how outputs are produced, which visible evidence supports outputs,
which visible rules support reasoning, and which declared boundaries constrain
interpretation. It also records which stage families and outputs are
implemented and how those outputs are inspected.

The reflexive state is `Framework Self-Description Available`. This means only
that the implemented methodology can describe itself from declared definitions
and outputs.

## Limitations

Stage 20 does not determine truth, liability, intent, blame, factual
correctness, legal sufficiency, or real-world sufficiency. It does not validate
evidence or create evidence, rules, conditions, classifications, methodology,
or framework identity. It does not introduce new reasoning, perform
self-improvement, or change framework behaviour. It describes only what already
exists within the implemented framework.

## Deterministic Constraints

Stage 20 is deterministic, read-only, reproducible, inspectable, non-mutating,
and backward compatible. Identical implemented definitions produce identical
self-description dictionaries. No external source, AI-generated narrative,
probabilistic inference, or external data retrieval is used.

It introduces no database writes, schema changes, migrations, background jobs,
public route changes, upload/download changes, canonical hash changes, or
attachment hash changes.

## Admin-Only Visibility

Framework Self-Description & Reflexive Methodology renders only in the Admin
Record Evidence view after Stage 19G Explainability Certification. Stage 20 adds
no public endpoint and changes no public visibility.

## Non-Mutating Behavior

Stage 20 returns newly constructed dictionaries and lists from fixed implemented
definitions. It does not modify records, versions, attachments, evidence,
classifications, existing framework outputs, hashes, or canonical
serialization.

## Expected Fixture Behavior

Every admin record fixture renders the same framework-level self-description:
the CREF Stage 20 identity, eleven implemented architecture layers, eight
available guarantees, sixteen visible constraints, seven ordered
self-description path steps, and `Framework Self-Description Available`.
